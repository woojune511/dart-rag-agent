# v0.2.0 Portfolio Ready

Release date: 2026-06-10

This checkpoint freezes the current portfolio submission baseline before
expanding benchmark experiments. It packages the repo as an applied
systems/research-engineering project: a contract-driven Agentic RAG runtime for
numeric QA over Korean DART filings.

The claim remains intentionally narrow. This release does not claim a new model
architecture, a general TableQA SOTA result, or complete elimination of
hallucination. It claims that financial RAG answers should be accepted through
inspectable runtime state: structured evidence, operand provenance,
deterministic calculation traces, critic reports, and reproducible gates.

## Scope

- Financial-document RAG runtime over Korean DART filings.
- Section/table-aware retrieval and value-cell-first structured metadata.
- Typed multi-agent artifact ledger:
  `tasks`, `artifacts`, `evidence_pool`, `critic_reports`, and
  `task_artifact_trace`.
- Numeric answer state through `answer_slots`, `structured_result`, and
  `resolved_calculation_trace`.
- Deterministic arithmetic, unit handling, dependency binding, validation, and
  final rendering after LLM-assisted semantic planning.
- Reviewer-facing documentation for README, one-pager, interview narrative,
  resume snippets, experiment report, technical highlights, and demo
  walkthrough.

## Closed Structural Ablation Baseline

The current portfolio ablation is deliberately small and closed.

| Variant | Numeric result | Interpretation |
| --- | ---: | --- |
| Structural full-system | `4/4` PASS | Preserves structural provenance and dependency operands through final calculation. |
| Plain retrieval counterpart | `3/4` PASS | Fails the separating case through unit/provenance drift. |

Separating case:

- `POS_T1_057`
- Plain retrieval preserves only one expected value and renders the
  operating-profit operand at the wrong scale.
- The structural path keeps both expected operands through final calculation.

Supported claim:

> Structured provenance plus dependency-operand preservation reduces final
> operand/unit drift when relevant numeric evidence is available but can be
> rebound or rendered at the wrong scale.

Unsupported claims:

- structure helps every DART question;
- the baseline is a comprehensive production competitor;
- the result establishes SOTA TableQA performance.

## Operational Boundary

This checkpoint does not promote disabled capabilities. The following remain
intentionally disabled:

- cache serving;
- retrieval bypass;
- automatic cache writes;
- cache candidate insertion into the live task/artifact ledger;
- LLM critic as final acceptance authority;
- benchmark-specific runtime routing branches.

Report-cache work remains candidate-only. `REFERENCE_NOTE` remains Researcher
graph-expansion context, not retrieval bypass or final acceptance authority.

## Artifact Hygiene

- Raw `benchmarks/results/**` bundles remain local experiment artifacts and are
  not part of the source checkpoint.
- The closed structural ablation profiles are committed, but local result
  directories are ignored.
- New benchmark runs should use monitored heartbeat logs when runtime can exceed
  a few minutes.
- Store-fixed `eval-only` refreshes should be preferred before paying for fresh
  ingest unless parser/ingest/cache signatures changed.

## Next Experiment Plan

The next experiment should expand only after this checkpoint is preserved.

Recommended order:

1. Expand the closed structural ablation from `4` questions to a small
   `8-12` question slice.
2. Keep the same variant pair first: structural full-system vs plain retrieval
   counterpart.
3. Record pass rate, failed question IDs, failure taxonomy, latency, retrieval
   query count, LLM call count, token use, and estimated runtime cost.
4. Classify failures as retrieval miss, wrong operand, unit drift, verifier
   miss, rendering drift, or evaluator ambiguity.
5. Update the experiment report only if the expanded result changes or
   strengthens the current claim boundary.

Do not add runtime branches for specific companies, benchmark IDs, or metric
phrases to improve the expanded experiment.
